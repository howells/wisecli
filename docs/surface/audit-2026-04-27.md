## Findings by Cluster

### Cluster 1: Structured Error Enrichment
*Error Handling (1→2) + Authentication (1→1.5)*  
Three dimensions share the same root fix: the `@howells/cli` error envelope is structurally thin. An agent receiving `{ok: false, error: "Wise API 429: ..."}` cannot decide to retry without parsing free text.

| Sev | File | Issue | Dimension |
|-----|------|-------|-----------|
| High | `src/api.ts:44-46` | HTTP error status embedded in string; 401/429/5xx indistinguishable | Error Handling |
| High | `@howells/cli` error output | No `is_retriable`, `code`, `detail` fields on error envelope | Error Handling |
| Medium | `src/index.ts` (all catch blocks) | All errors exit 1; no semantic exit codes (77=auth, 69=unavailable, 2=misuse) | Error Handling |
| Low | `src/api.ts` | `Retry-After` header from Wise 429 discarded; no `retry_after_seconds` in output | Error Handling |
| Low | `src/accounts.ts:55-61` | First-token fallback is non-deterministic when >1 token configured | Authentication |

**Suggested approach:** Create a custom `ApiError` class in `src/api.ts` that captures `status`, `is_retriable`, and `retry_after_seconds`. Extend the `@howells/cli` error envelope to include `code` (enum string), `is_retriable` (bool), and `detail`. Map HTTP status codes to semantic exit codes in `src/index.ts` catch blocks.

### Cluster 2: MCP Server + Tool Adapter Layer
*MCP Server (0→2+) + Tool Design (1→2+)*  
The CLI's well-typed functions (`balanceForAllProfiles`, `transfers`, `listProfiles`, `resolveAccount`) are perfect MCP tool candidates. The schema command already has the metadata. The gap is the adapter layer that exposes them to agent runtimes.

| Sev | File | Issue | Dimension |
|-----|------|-------|-----------|
| Critical | (absent) | No MCP server file, no `@modelcontextprotocol/sdk` | MCP Server |
| Critical | (absent) | No `pnpm add @modelcontextprotocol/sdk zod` | MCP Server |
| High | `src/index.ts:159-229` | Command descriptions terse (<20 words, no "Use when/not" patterns) | Tool Design |
| High | `src/index.ts:171-229` | Most params lack `description` and `example` fields in schema output | Tool Design |
| Medium | `src/index.ts:234` | `status` param has no enum; Wise statuses are enumerable | Tool Design |
| Medium | (absent) | No MCP `readOnlyHint: true`, `idempotentHint: true` annotations | Tool Design |
| Low | `src/commands.ts:134-157` | No `has_more`/`next_cursor` in transfer responses; agents can't paginate | Tool Design |

**Suggested approach:** Create `src/mcp-server.ts` with `StdioServerTransport`; wrap each command function in a `server.tool()` call reusing existing TypeScript functions directly. Generate Zod schemas from the existing `schema` command metadata. Expand descriptions to 60–120 words with "Use when / Do not use for" patterns. Add `readOnlyHint: true` to all tools.

*Depends on: Cluster 1 (error envelope needed for MCP isError responses)*

### Cluster 3: Context & Discovery Layer
*Context Files (2→3) + Discovery & AEO (2→3)*  
The AGENTS.md is strong — it just needs three additions to complete the context layer. llms.txt and llms-full.txt are low-effort and high-impact for agents that ingest the package directory at install time.

| Sev | File | Issue | Dimension |
|-----|------|-------|-----------|
| High | (absent) | No CLAUDE.md (even a thin one gates Context Files score 3) | Context Files |
| Medium | `AGENTS.md` | No explicit Always/Ask-first/Never permission tiers | Context Files |
| Medium | (absent) | No `llms.txt` at repo root (applicable to CLI npm packages) | Discovery & AEO |
| Medium | (absent) | No `llms-full.txt` (concatenated README + AGENTS.md + schema output) | Discovery & AEO |
| Low | `AGENTS.md` | No testing/build commands (`pnpm test`, `pnpm typecheck`) | Context Files |
| Low | `AGENTS.md` | No source layout section referencing `src/` module paths | Context Files |

**Suggested approach:** Add a minimal `CLAUDE.md` referencing AGENTS.md. Add a `## Permissions` section to AGENTS.md. Generate `llms.txt` and `llms-full.txt` at repo root and include both in `package.json` `files` array.

### Cluster 4: Agent Testing
*Testing (0→1+)*  
The 5 unit test files are well-written but test the HTTP/data layer in isolation. No test verifies that an agent can correctly invoke the CLI or select the right command from a natural-language prompt.

| Sev | File | Issue | Dimension |
|-----|------|-------|-----------|
| High | (absent) | No `.github/workflows/` CI pipeline; tests never run automatically | Testing |
| High | (absent) | No tool-routing or CLI invocation tests (spawn binary, verify command selection) | Testing |
| Medium | (absent) | No MCP InMemoryTransport tests (blocked on Cluster 2) | Testing |
| Medium | `package.json` | No coverage thresholds configured | Testing |

**Suggested approach:** Add `ci.yml` GitHub Actions workflow with `pnpm test` on push/PR. Add `src/__tests__/cli-integration.test.ts` that spawns the binary via `execa` and asserts on JSON output shapes. After Cluster 2, add `src/__tests__/mcp-server.test.ts` using InMemoryTransport.

*Depends on: Cluster 2 for MCP tests*

---

### Cluster 5: Authentication Documentation Hardening
*Authentication (1→1, documentation-only)*  
The Wise API doesn't expose OAuth 2.1 M2M (upstream constraint), so the score ceiling is limited. The remaining improvements are documentation and operational resilience.

| Sev | File | Issue | Dimension |
|-----|------|-------|-----------|
| Medium | `README.md`, `AGENTS.md` | No guidance to create read-only tokens at wise.com/settings/api-tokens | Authentication |
| Low | `src/api.ts` | 401 errors give no token-rotation guidance | Authentication |
| Low | `src/accounts.ts:55-61` | First-token fallback non-deterministic when multiple env vars configured | Authentication |

**Suggested approach:** Add one sentence to README and AGENTS.md: "When creating your token, select Read-only access." On 401, include `"hint": "Token may be invalid or revoked. Check WISE_<NAME>_TOKEN at wise.com/settings/api-tokens."` in the error envelope.
